## 1. Project Vision & Moat

LearnAI is an **AI-powered adaptive exam-preparation platform**. Users upload study notes; the platform generates personalized interactive lessons; the learner's ongoing performance is continuously modelled and fed back to them as an honest picture of exam-readiness.

### The Moat: Honest, Behaviour-Derived Learner Modelling

The core competitive differentiator is that **every readiness and mastery signal is derived exclusively from objectively-gradeable, observed behaviour** — never from self-reported learning styles, self-assessed skill level, or any other subjective proxy.

| What we track | Where it comes from | What we never use |
|---|---|---|
| Per-concept mastery (Beta posterior) | Graded quiz first-attempts | Self-reported confidence as mastery |
| Exam-readiness % | Importance-weighted mastery rollup | Course-completion % |
| Confidence calibration gap | Pre-submit self-rating vs actual accuracy | "How confident do you feel overall?" |
| Prerequisite-repair recommendations | Concept graph edges + weak mastery | Learning style questionnaires |
| Mistakes queue | Wrong first-attempts (FK-anchored) | Subjective "I found this hard" |

**Honesty invariant**: Every first-graded-attempt carries a durable `is_first_attempt = true` marker. A partial unique index `(user_id, step_id) WHERE is_first_attempt` enforced at the database level makes it physically impossible for two concurrent submissions to both be counted as first attempts for the same step. Mastery updates, mistake creation, and the global readiness gate all read from this durable marker — never from an in-memory count that a race condition could corrupt.

## 7. Learner Model — Science & Implementation

### 7.1 `GET /dashboard/learner-model` Response Shape

```typescript
{
  examReadiness: number | null,      // 0-100; concept-weighted when sufficient data
  masteryLevel: string | null,       // "strong" | "proficient" | "developing"
  confidence: number,                // 0-1; how much evidence backs the score
  accuracy: number,                  // raw correct / total graded (all attempts)
  selfReliance: number,              // 1 − (hintsUsed / max(1, totalGraded))
  signals: Signal[],
  strengths: string[],
  focusAreas: string[],
  calibration: {
    score: number,                   // 0-100; 100 = perfect calibration
    direction: "overconfident" | "calibrated" | "underconfident",
    avgConfidence: number,
    sampleSize: number
  } | null,
  conceptMastery: ConceptMastery[],  // per-concept detail
  readinessByCourse: CourseReadiness[],
  prerequisiteRepairs: { concept: string; prerequisite: string }[]
}
```

### 7.2 Exam-Readiness Score

```
readiness = round(100 × Σ(importance × mastery × gate) / Σ(importance × gate))
```

where:
- `importance` — concept importance weight (default 1.0, set at extraction time)
- `mastery` — Beta posterior mean: α / (α + β)
- `gate` = `min(1, firstAttempts / 5)` — evidence gate; a concept with no observations contributes nothing to the denominator, preventing the prior from inflating readiness

**Unlock condition**: The concept-level rollup replaces the aggregate estimate only when `COUNT(DISTINCT first-attempt answer events) >= 5`. This count comes from the `is_first_attempt` column (not from per-concept counters, which would double-count multi-concept questions).

**Fallback**: When fewer than 5 first-attempts exist, readiness falls back to `round(100 × (0.7 × accuracy + 0.3 × selfReliance))`.

### 7.3 Per-Concept Beta-Binomial Mastery Update

Triggered **only** on first attempts (`is_first_attempt = true`). Retries are recorded in `answer_events` for accuracy tracking but never touch `mastery_records`.

```
diffWeight = { beginner: 0.8, intermediate: 1.0, advanced: 1.2 }[course.difficulty]
numConcepts = count of concepts tagged to this step

for each tagged concept c with weight w_c:
    evidence = diffWeight × w_c / numConcepts      # splits across concepts
    correctEvidence  += isCorrect ? evidence : 0
    incorrectEvidence += isCorrect ? 0 : evidence
    α += correctEvidence
    β += incorrectEvidence
    mastery    = α / (α + β)                       # posterior mean
    confidence = min(1, firstAttempts / 5)         # evidence gate
```

**Priors**: α₀ = β₀ = 2 (Beta(2,2) — uninformative but slightly regressed toward 0.5, avoiding extremes at low evidence).

**Evidence splitting**: A step tagged to *n* concepts contributes exactly one unit of difficulty-weighted evidence total, divided equally (or by declared weight) across those concepts. This prevents a multi-concept question from inflating alpha for all concepts.

### 7.4 Mastery Bands

| Band | Threshold | Colour |
|---|---|---|
| Strong | mastery ≥ 0.80 | Green |
| Proficient | mastery ≥ 0.60 | Amber |
| Developing | mastery ≥ 0.40 | Sky |
| Weak | mastery < 0.40 | Red |

### 7.5 Confidence Calibration

Every graded answer carries a pre-submit self-rating (`confidence` 0-100) recorded in `answer_events`. After 5+ rated answers:

```
avgConfidence = mean(confidence) / 100
accuracy      = mean(isCorrect)
gap           = avgConfidence − accuracy          # >0 overconfident, <0 underconfident
score         = round((1 − min(1, |gap|)) × 100)
direction     = gap > 0.10 → "overconfident"
              | gap < −0.10 → "underconfident"
              | else → "calibrated"
```

Standard pre-submit ratings: "Just guessing" = 25, "Fairly sure" = 60, "Certain" = 90.

Calibration is **independent** of the readiness gate — it is computed from all events regardless of whether concept-level rollup has unlocked.

### 7.6 Prerequisite Repair

The concept graph (`concept_edges`) encodes `prerequisite → dependent` relationships extracted by the AI. After computing mastery, the learner-model endpoint traverses edges: if a dependent concept is weak (`mastery < 0.40`) **and** its prerequisite is also weak, the repair is surfaced:

> "Strengthen `{prerequisite}` before tackling `{dependent}`"

Up to 2 repairs are returned, prepended to `focusAreas`.

### 7.7 Concurrency Integrity

The transaction in `progress.ts` submit_answer:

1. `SELECT … FOR UPDATE` on the `course_progress` row — serialises concurrent submissions for the same (user, course).
2. Checks `COUNT(answer_events WHERE user_id=? AND step_id=?)` **before** inserting — if 0, this is the first attempt.
3. Inserts `answer_events` with `is_first_attempt` set appropriately.
4. The partial unique index `(user_id, step_id) WHERE is_first_attempt` is the database-level backstop: even if the application logic were bypassed, the DB would reject a second `is_first_attempt = true` insert for the same (user, step).
5. Counter updates to `course_progress` use atomic SQL increments (`correctAnswers + 1`), never stale in-memory values.

## 8. AI Generation Pipeline

### 8.1 Course Step Generation (`POST /courses/:id/generate-steps`)

**Model**: `gpt-5.4`  
**Transport**: Server-Sent Events (`text/event-stream`)

The prompt instructs the model to return a structured JSON object containing:

```json
{
  "steps": [
    {
      "type": "content|quiz|code_exercise",
      "title": "...",
      "content": "...",
      "concepts": ["Concept A", "Concept B"],
      "questionData": { "question": "...", "correctAnswer": "...", "explanation": "...", "options": [...] }
    }
  ],
  "concepts": [
    { "title": "...", "description": "...", "importance": 1.0 }
  ],
  "edges": [
    { "prerequisite": "Concept A", "dependent": "Concept B" }
  ]
}
```

**Persistence pipeline** (idempotent — safe to retry):

1. Wipe existing concepts for the course (cascade clears `concept_edges`, `lesson_step_concepts`).
2. Insert concepts, building a case-insensitive `name → id` map.
3. Insert edges; skip self-loops, unknown concept references, and duplicates (`onConflictDoNothing`).
4. For each step, insert `lesson_step_concepts` linking the step to its tagged concepts.
5. Emit `concepts_extracted` SSE event.
6. Emit a `step` SSE event for each generated step.
7. Emit `done`.

### 8.2 AI Hint (`POST /…/hint`) — SSE

Context: course title, step content, question. Instructs GPT not to give the answer — only a guiding nudge.

### 8.3 AI Explanation (`POST /…/explain`) — SSE

Context: course title, step content. Full conceptual elaboration.

### 8.4 AI Tutor Chat (`POST /openai/conversations/:id/messages`) — SSE

Full conversational tutoring. System prompt includes course context. Streams token-by-token.

## 14. Key Architectural Decisions & Gotchas

### Contract-First API

All API changes start with `openapi.yaml`. Codegen produces the typed hooks and schemas. Manually written types that duplicate the spec are a code smell — use generated types.

### No `console.log` in Server Code

Use `req.log` (Pino child logger, carries request context) inside handlers. Use the singleton `logger` for non-request code. `console.log` bypasses structured logging and won't appear in the correct log stream.

### Express 5 Param Typing

`req.params.id` is typed `string | string[]` in Express 5. Always access as `req.params["id"] as string`.

### API Hooks Expect `number` IDs

Generated hooks accept `number` for `id` params. Use `Number(id)` or pass the integer directly — never pass a string.

### SSE Endpoints

Always set:
```
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
```

### DB Schema Exports

Every new table must be added to `lib/db/src/schema/index.ts` for `@workspace/db` to expose it. Missing exports cause `Cannot find name` TS errors in leaf packages.

### Migrations on Replit

`pnpm --filter @workspace/db run push` (Drizzle Kit push) requires an interactive TTY and doesn't work from the agent sandbox. Apply schema changes via raw SQL using `executeSql()` in the code_execution sandbox instead.

### Composite Libs vs Leaf Artifacts

- `lib/*` packages: composite, emit declarations, listed in root `tsconfig.json` references.
- `artifacts/*`: leaf, `tsc --noEmit` only, **never** added to root `tsconfig.json`.
- Always run `pnpm run typecheck:libs` before `pnpm --filter @workspace/api-server run typecheck`.

### Unique `(user_id, course_id)` on `course_progress`

Without this, two concurrent requests could create two progress rows for the same (user, course), causing mastery and accuracy aggregates to diverge.

### Atomic Counter Increments

`course_progress` counters (`correct_answers`, `incorrect_answers`, `total_xp`) are updated via `column + N` SQL expressions — never `SET correct_answers = <stale in-memory value>`.

### `is_first_attempt` Partial Unique Index

The most critical correctness invariant. The partial unique index `ON answer_events (user_id, step_id) WHERE is_first_attempt` is the database-level guarantee that the honesty invariant cannot be violated by concurrent requests, application bugs, or retries.

## 15. Roadmap — Phases 2 & 3

### Phase 2 — Tasks Page + Spaced Repetition + Retry Mistakes

**Goal**: surface an actionable daily task queue so the learner always knows exactly what to do next.

| Feature | Description |
|---|---|
| FSRS-lite scheduling | On every `mastery_records` update, compute next review interval using SM-2/FSRS formula; write `review_schedules.due_at` |
| `GET /tasks` | Returns today's due reviews + open mistakes, ordered by priority (overdue first, then new mistakes) |
| `POST /tasks/:conceptId/review` | Submit a recall rating; update `review_schedules` (stability, difficulty, retrievability); update `mastery_records` |
| Tasks page | Dedicated `/tasks` page in the nav; shows due reviews as flashcard-style UI; shows open mistakes as retry queue |
| Nav badge | Count of overdue items in sidebar |

### Phase 3 — Real Uploads + OCR + Source Grounding

**Goal**: accept PDF and image notes; ground AI-generated content in the actual source so the learner can verify claims.

| Feature | Description |
|---|---|
| Object storage | Store uploaded files in Replit Object Storage; record `file_url` + `mime_type` in `notes` |
| PDF parsing | Extract text from PDFs server-side (pdfjs-dist or similar) |
| OCR for images | Send image to GPT-4o Vision to extract text |
| `source_chunks` table | Store extracted text chunks with `(note_id, chunk_index, text, embedding vector)` |
| pgvector | Enable the `vector` extension; add `embedding vector(1536)` to `source_chunks`; index with `ivfflat` |
| Grounded generation | Before generating steps, retrieve top-k relevant chunks via cosine similarity; inject into the system prompt |
| "Not in notes" guard | If the learner asks the tutor something not covered in the source chunks (low similarity score), the tutor says so — preventing hallucination |
